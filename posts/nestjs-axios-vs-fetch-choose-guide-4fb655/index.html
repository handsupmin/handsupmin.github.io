NestJS에서 Axios vs Fetch 선택 가이드 — RxJS 통합 vs 표준 가벼움

December 3, 2025

개요#

NestJS 환경에서 Axios와 Fetch 선택은 RxJS 중심의 생태계 통합성 vs 표준 API의 가벼움 사이 트레이드오프 이슈임 Node.js 18 이상부터 fetch가 내장되면서 의존성 없이도 표준 HTTP 클라이언트를 쓸 수 있게 되었음

Axios — NestJS 권장 경로#

NestJS는 @nestjs/axios 기반 HttpModule을 제공하며 HttpService로 Observable 응답을 다루는 흐름을 권장함. Nest 애플리케이션 전반에서 RxJS를 쓰는 경우 자연스럽게 결합됨

장점

  • RxJS와 자연스러운 통합. HttpService 응답이 Observable 형태라 스트림 연산자(map, tap, catchError) 조합 용이
  • 인터셉터 지원 강력. 요청/응답 로깅, 공통 헤더 삽입, 에러 표준화 같은 횡단 관심사 구현이 쉬움
  • 편의 기능 제공. 자동 JSON 변환, 4xx/5xx 발생 시 예외 throw, 타임아웃과 요청 취소 지원
  • 테스트 용이성 우수. HttpService mocking이 쉬워 단위 테스트 작성 비용 낮음

단점

  • 외부 라이브러리 의존으로 런타임/번들 사이즈 증가. 백엔드에서는 보통 치명적 이슈 아님
  • Promise로 단순 처리하려면 firstValueFrom/lastValueFrom 변환 필요. 비동기 제어 단순화를 선호할 때 불편함 존재

예시 스니펫

import { HttpService } from '@nestjs/axios'
import { lastValueFrom } from 'rxjs'

async function getData(httpService: HttpService) {
  const res = await lastValueFrom(httpService.get('url'))
  return res.data
}

Fetch — Node.js 18+ 내장 표준 API#

별도 설치 없이 글로벌 fetch 사용 가능. 브라우저와 서버 간 동일한 모델로 코드를 공유하기 쉬움

장점

  • 의존성 없음. 표준 API라 장기 유지 가능성과 이식성 높음
  • Promise 기반의 단순한 사용성. async/await와 자연스럽게 결합됨
  • 동일한 fetch 인터페이스로 브라우저/서버 이소모픽 코드 구성 용이

단점

  • 보일러플레이트 증가. JSON 변환은 response.json() 호출 필요, HTTP 에러도 ok 체크 후 명시적으로 throw 처리 필요
  • NestJS DI에 직접 얹으려면 래퍼 서비스나 커스텀 프로바이더 필요. 테스트 시 mocking 고려까지 포함하면 작업량 증가
  • 기본 기능 부족. 인터셉터, 타임아웃, 요청 취소 등은 직접 구현 또는 추가 유틸 필요

예시 스니펫

async function getData() {
  const res = await fetch('url')
  if (!res.ok) throw new Error('Failed')
  return await res.json()
}

한눈에 비교#

  • 설치와 의존성
    • Axios: 추가 설치 필요
    • Fetch: Node 18+ 내장
  • 리턴 타입
    • Axios: Observable 중심. RxJS 친화적
    • Fetch: Promise 중심. async/await 간결
  • JSON 파싱
    • Axios: 자동 변환
    • Fetch: response.json() 수동 호출
  • 에러 처리
    • Axios: 4xx/5xx에서 자동 throw
    • Fetch: res.ok 검사 후 수동 throw 필요
  • 인터셉터 및 횡단 관심사
    • Axios: 인터셉터 제공. 로깅/인증/재시도 패턴 구현 용이
    • Fetch: 직접 구현 필요
  • NestJS 통합과 테스트
    • Axios: HttpModule/HttpService 기반. DI와 mocking 수월
    • Fetch: 래퍼 작성 시 테스트 용이성 확보 가능하나 초기 작업 필요

선택 가이드#

다음 조건이면 Axios 추천

  • NestJS에서 RxJS 스트림 처리 활용 비중 높음
  • 로깅, 인증 헤더 자동 주입, 에러 표준화 등 인터셉터 기반 공통 처리 필요
  • 팀이 Nest 패턴과 @nestjs/axios 모듈에 익숙하거나 테스트 작성 비중이 큼

다음 조건이면 Fetch 추천

  • 외부 호출이 적고 간단한 서비스나 사이드 프로젝트
  • Promise 중심으로 최소 의존성과 단순한 호출 흐름 선호
  • 브라우저와 서버 간 이소모픽 코드 재사용 우선

정리#

대부분의 NestJS 서비스에서는 @nestjs/axios 채택이 운영 편의와 일관성 측면에서 유리함. 반면 경량 서비스나 의존성 최소화가 핵심이면 내장 fetch가 합리적 선택임. 팀의 RxJS 숙련도, 공통 인터셉터 요구, 테스트 전략을 기준으로 결정하는 것이 실수 최소화에 도움이 됨
