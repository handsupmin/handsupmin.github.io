전략 패턴으로 행위를 동적으로 교체하기 — OCP 관점과 Java 활용 포인트

December 6, 2025

개요#

전략 패턴은 객체의 행위를 직접 수정하지 않고, 행위를 캡슐화한 전략 객체를 갈아끼우는 방식으로 동작을 교체하는 설계 방식임 핵심은 상속보다 합성 위주 설계와 인터페이스 기반 위임이며, 런타임에 전략을 바꾸어도 클라이언트 코드는 안정적으로 유지 가능함

왜 필요한가#

버스와 기차 같은 운송 수단이 있고 move라는 동일 행위를 제공한다고 가정 처음에는 버스는 도로, 기차는 선로를 따른다고 구현했다고 하자 이후 선로를 따라 움직이는 버스 요구가 생기면 기존 Bus의 move 구현을 직접 바꾸게 됨 이는 OCP(Open-Closed Principle) 위배 가능성 높음

  • 기존 클래스 내부 구현 수정 필요 발생
  • 유사 클래스에 동일 변경이 중복됨
  • 확장 방향에 따라 메서드 중복과 결합도 상승

전략 패턴은 행위를 전략 인터페이스로 분리하여 이러한 변경을 클래스 내부 수정 없이 전략 교체로 흡수하도록 유도함

핵심 개념#

  • Strategy 인터페이스: 행위의 공통 계약 정의, 예) MovableStrategy의 move 정의
  • Concrete Strategy: 구체 행위 구현, 예) RailRoad 전략, Road 전략
  • Context: 전략을 보유하고 실행 시 위임, 예) Moving이 setStrategy로 전략 주입 후 move 시 전략 호출
  • Client: 상황에 맞게 전략을 선택·교체, 런타임 교체 가능

전략의 포인트는 행위 변경을 상속 트리 변경이 아니라 전략 객체 교체로 처리하는 것임

동작 원리#

  • 행위 축을 인터페이스로 분리하여 캡슐화
  • Context는 구체 행위를 알지 않고 인터페이스에만 의존
  • Client는 요구사항 변화에 맞춰 전략을 생성해 Context에 주입
  • 런타임에 set으로 전략 변경 가능, 기존 클래스의 함수 본문 수정 없이 동작 변경 달성

전략 확장도 독립적임

  • 새로운 경로 전략 추가 시 기존 Context·클라이언트 코드는 타입 계약만 유지하면 영향 최소화
  • 다른 축의 전략도 병렬로 추가 가능, 예) 연료 전략, 요금 정책 전략 등

간단 예시#

필요 최소 스니펫만 제시

public interface MovableStrategy {
    void move();
}

public class Moving {
    private MovableStrategy strategy;
    public void setMovableStrategy(MovableStrategy s) { this.strategy = s; }
    public void move() { strategy.move(); }
}

전략 교체 예시

Moving bus = new Moving();
bus.setMovableStrategy(new RoadStrategy());
bus.move(); // 도로 이동

bus.setMovableStrategy(new RailRoadStrategy());
bus.move(); // 선로 이동

핵심은 Moving의 코드 변경 없이 버스의 이동 행위를 교체 가능하다는 점임

주의와 베스트 프랙티스#

  • 전략 수 폭증 관리 필요

    • 전략 클래스가 많아지면 생성·선택 규칙이 복잡해질 수 있음
    • Factory 또는 DI 컨테이너와 조합하여 전략 선택 책임을 분리 권장

  • 기본 전략 설정

    • 널 전략 사용 또는 합리적 기본값 주입으로 NPE 방지

  • 전략 선택 기준의 외부화

    • if-else 분기가 클라이언트 곳곳에 흩어지지 않도록 매핑 규칙을 구성으로 분리
    • 환경 설정, 피처 플래그, 정책 테이블로 전략 선택을 데이터화

  • 인터페이스 과도 분해 주의

    • 전략 축이 명확하지 않으면 오히려 복잡도 증가
    • 변경 가능성이 높은 행위만 전략화

  • 성능 고려

    • 호출 경로가 잦은 고빈도 구간에서는 전략 호출 오버헤드와 객체 수 고려
    • 필요한 경우 전략을 불변·싱글턴화하여 할당 비용 절감

Java에서의 활용 포인트#

  • Comparator는 대표적인 전략의 예로, Collections.sort가 비교 전략에 위임
  • Servlet의 service 및 doXXX 계열은 요청/응답 처리 전략을 구현체가 주도
  • Servlet Filter의 doFilter는 체인 전략 조합으로 동작 위임

표준 API가 전략을 통해 개방-폐쇄를 실무적으로 달성하는 패턴을 보여줌

마무리#

전략 패턴은 변경 가능성이 높은 행위를 인터페이스로 분리하고 합성으로 조립하여, 행위 변경을 클래스 수정이 아닌 전략 교체로 해결하는 기법임 OCP 준수, 중복 감소, 테스트 용이성 향상이라는 이점을 제공하나 전략 수 관리와 선택 규칙 외부화 같은 운영 상의 설계 포인트가 존재함 요구 변화가 잦은 행위 축을 식별했다면 우선 전략 분리와 주입을 고려할 가치가 큼
