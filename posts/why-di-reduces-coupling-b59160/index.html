DI가 결합도를 낮추는 원리와 최소 예시

November 9, 2025

개념/배경#

DI(의존성 주입)의 요지는 간단함 객체가 스스로 의존 대상을 생성하지 않고 외부에서 받은 것을 사용함 이 작은 차이가 변경 용이성, 테스트 편의, 확장성에서 큰 차이를 만듦

문제 상황: 강한 결합#

문제 패턴

  • 클래스 내부에서 new로 구체 클래스를 직접 생성함
  • 변경 시 내부 코드를 같이 수정해야 함
  • 단위 테스트 시 실제 의존체까지 끌려와 테스트가 무거워짐

예시 요약

  • Gamer가 BlueSwitchKeyboard를 생성해 사용한다고 가정
  • Gamer는 BlueSwitchKeyboard라는 구체 구현에 고정됨
  • 키보드 종류를 바꾸려면 Gamer 코드를 수정해야 함 이 상태를 결합도가 높음이라 부름

간단 스니펫

class Gamer {
  constructor() {
    this.keyboard = new BlueSwitchKeyboard()
  }
  play() {
    this.keyboard.type()
  }
}

DI 적용: 느슨한 결합#

핵심 변경점

  • Gamer는 키보드를 만들지 않음
  • 외부에서 주입받은 keyboard에만 의존함
  • 계약은 type 메서드를 가진다는 점 하나로 축약

간단 스니펫

class Gamer {
  constructor(keyboard) {
    this.keyboard = keyboard
  }
  play() {
    this.keyboard.type()
  }
}

이제 어떤 키보드든 type만 제공하면 교체 가능함

교체 예시 요약

  • BlueSwitchKeyboard, RedSwitchSilentKeyboard 둘 다 주입 가능
  • Gamer 코드는 변경 없음

왜 결합도가 낮아지는가#

의존 대상이 구체 구현에서 역할로 전환됨

  • before: Gamer → BlueSwitchKeyboard라는 특정 구현 결합
  • after: Gamer → type을 제공하는 키보드 역할에 의존 결합 축이 구현이 아닌 계약으로 이동하면서 변경 파급이 줄어듦

테스트와 확장성#

테스트 용이성

  • 실제 하드웨어 의존 제거 가능
  • 가짜 구현으로 시나리오를 빠르게 구성 가능

간단 스니펫

const mockKeyboard = { type: () => console.log('mock type') }
const g = new Gamer(mockKeyboard)
g.play()

확장성

  • 새로운 키보드가 추가되어도 Gamer 수정 불필요
  • 조립부에서 어떤 구현을 줄지 결정하면 됨

DI와 IoC 컨테이너#

수동 DI만으로도 효과 충분함 규모가 커지면 의존성 생성과 수명 관리가 반복됨 프레임워크의 IoC 컨테이너는 생성과 주입을 중앙에서 관리함

  • 생성자에 필요한 것을 선언하면 컨테이너가 인스턴스를 주입함
  • 범위 스코프, 라이프사이클, 순환참조 검출 등의 부가 기능 제공 가능

주의사항#

  • 서비스 로케이터 남용 주의, 의존성을 숨기는 패턴이라 테스트와 추론성 저하 위험
  • 생성자 인자가 과도하게 늘어나면 설계 재검토 필요, 책임 분리 고려
  • 런타임 타입 체커가 없는 환경에서는 계약 위반을 조기에 놓치기 쉬움, 타입스크립트 또는 런타임 검증 도입 고려
  • 순환 의존성은 구조 문제 신호, 역할 분리 또는 중간 포트 도입으로 끊기

마무리#

DI의 본질은 생성 책임과 사용 책임을 분리하는 일임 사용자는 역할 계약만 바라보고 동작하고, 생성은 외부 조립부나 컨테이너가 담당함 그 결과 구현 교체가 쉬워지고 테스트가 가벼워지며 결합도가 낮아짐
