AWS Lambda의 팬아웃·팬인 패턴과 NestJS·Redis로 현재 파드 전부에 안전한 팬아웃 구현

November 10, 2025

개념/배경#

fan-out/fan-in은 push-pull 계열 메시징 패턴을 조합한 형태로 보는 게 정확함

  • fan-out: 메시지를 여러 워커에 분산, 각 메시지는 한 워커가 처리하는 경쟁 소비자 모델 혹은 높은 병렬 처리 목표의 브로드캐스트 계열 사용
  • fan-in: 분할된 서브태스크의 결과를 수집·집계하여 상위 작업의 완료를 도출

서버리스에서 fan-out은 처리량과 동시성 확보, fan-in은 결과 집계와 진행 추적을 위한 상태 관리가 핵심 포인트임

참고로 현재 Lambda 최대 실행 시간은 15분임. 과거 5분 제한 기준의 논의와 다를 수 있음

SNS로 팬아웃#

목표가 빠른 고병렬 팬아웃이면 SNS가 단순하고 확장 쉬움

  • ventilator 함수가 고비용 작업을 여러 서브태스크로 분할 후 SNS 토픽에 퍼블리시
  • 구독된 Lambda 워커가 메시지 단위로 병렬 실행되어 처리량 상승
  • 게시글을 팔로워 타임라인에 전파하는 유형의 업무에서 팔로워를 청크로 나눠 병렬 처리하는 식으로 적용 가능

특성 및 유의

  • at-least-once 전달 특성, 워커는 멱등성 필요
  • 서브태스크가 크면 S3 등에 페이로드 저장하고 SNS엔 레퍼런스 전달 고려
  • 구독 수 만큼 브로드캐스트되므로 동일 코드를 여러 구독으로 중복 연결하면 전체 메시지를 모두 수신하게 됨

SQS로 팬아웃#

현재는 SQS가 Lambda 트리거를 지원하므로 경쟁 소비자 모델 구현에 적합함

  • 메시지 가시성 타임아웃, 재시도, DLQ 제어로 내구성과 흐름 제어 확보
  • 서브태스크가 상대적으로 오래 걸리거나 백프레셔가 필요한 워크로드에 유리

운영 포인트

  • 가시성 타임아웃은 처리 시간 + 여유를 반영하여 설정, 실패 시 DLQ로 유실 방지
  • 메시지 중복 가능성 존재, 워커 멱등성 필수
  • 과거엔 재귀 Lambda + SQS 핸들로 가시성 연장 등의 우회가 필요했으나, 오늘날에는 표준 트리거 조합으로 단순화 가능

Kinesis와 DynamoDB Streams는 언제#

이벤트 스트림은 병렬성이 샤드 수에 묶이는 구조라 대규모 팬아웃에는 덜 적합한 경우가 많음

  • Kinesis는 샤드 기반 병렬성, 리샤딩 비용과 운영 복잡도 존재. 온디맨드 또는 프로비저닝 선택과 보존 설정에 따른 비용 고려 필요
  • DynamoDB Streams는 관리형 샤딩으로 직접 제어가 어려움

대신 다음 상황에서는 유리함

  • 샤드 단위 순서 보장 필요
  • 테이블 변경 캡처 기반 파이프라인 구성

팬인: 결과 수집 전략#

ventilator가 각 서브태스크에 상위 jobId와 taskId를 포함해 발행, 워커는 완료 시 식별자로 결과 기록

저장소 선택

  • DynamoDB 결과 저장. 항목 최대 400KB 제한, 메타데이터와 경량 결과에 적합
  • S3 결과 저장. 객체 최대 5TB, 대용량 결과에 적합. 예시 키 구조

핫 파티션/프리픽스 완화

  • jobId를 GUID로 생성하여 균등 분산
  • 상위 디렉터리 해시 분산 프리픽스 도입
  • S3에 무작위 키로 저장하고 DynamoDB에는 참조만 기록하는 이중 저장 패턴 적용

팬인: 전체 진행 추적#

ventilator가 총 서브태스크 수를 기록, 각 워커가 원자적으로 남은 수를 감소. 0이 되는 전환에서 전체 완료 신호 발행

구현 힌트

  • DynamoDB UpdateExpression으로 원자 감소와 0 도달 감지
  • remaining이 0이면 SNS 등으로 완료 이벤트 발행, 후속 집계 sink 트리거
  • 결과가 매우 많으면 sink도 재귀 또는 배치 처리로 분할. 멱등성 키로 중복 집계 방지

운영 주의

  • 부분 실패 대비 재시도 시 중복 집계를 피하기 위해 작업 단위 idempotency 키 유지
  • 서브태스크 타임아웃과 리트라이 정책을 전체 SLA에 맞게 조정

NestJS + Redis로 현재 파드 전부에 팬아웃#

요구 정의

  • 지금 Running 상태의 모든 파드가 해당 메시지를 반드시 수신해야 함

선택지 요약

  • A. Redis Pub/Sub: 구독 중인 파드에 실시간 전송. 단순·저지연. 구독 경계에서 유실 가능
  • B. Redis Streams + 파드별 Consumer Group: at-least-once로 모든 파드가 반드시 수신. 운영 복잡도와 메모리 증가 감수

핵심 요구가 반드시 전달이면 B 권장

공통 준비

  • 파드 식별자: HOSTNAME 또는 POD_NAME를 그룹/컨슈머 이름으로 사용
  • Redis 연결은 퍼블리셔와 서브스크라이버를 분리 구성 권장
  • 레디니스는 구독 또는 스트림 폴링 시작 이후에만 Ready 전환하여 롤링 중 유실 방지

A) Pub/Sub

  • 퍼블리셔는 채널에 JSON 직렬화하여 publish
  • 각 파드는 부팅 시 subscribe 후 message 이벤트에서 처리

최소 스니펫

장단

  • 장점: 코드와 운영 단순, 초저지연
  • 한계: 구독이 끊긴 동안의 메시지는 복구 불가. 지금 구독 중인 파드 모두에게만 보장

B) Streams + 파드별 Consumer Group

  • 스트림에 XADD 한 번으로 모든 파드가 속한 각자 그룹이 동일 메시지를 수신해야 ACK하도록 설계
  • 파드 시작 시 XGROUP CREATE로 pod: 그룹을 만들고 XREADGROUP으로 BLOCK 폴링, 처리 후 XACK
  • 메시지 보존과 메모리 관리를 위해 XTRIM 설정 권장

최소 스니펫

장단

  • 장점: 파드 재시작·일시 단절 상황에서도 미처리 엔트리는 Pending으로 남아 재전달, 각 파드 최소 한 번 수신 보장
  • 한계: 파드 수만큼 그룹이 생겨 메시지당 N번의 전달·ACK 수행. 대규모 스케일에서 메모리와 관리 비용 증가. 유휴 그룹 정리 필요

배포·운영 체크리스트

  • HOSTNAME 주입. Kubernetes Downward API로 환경변수 노출
  • readinessProbe는 구독 또는 스트림 루프 시작 이후 성공으로 전환
  • preStop 훅이나 주기적 GC로 유휴 그룹 정리. 예시 흐름: XINFO GROUPS로 빈 그룹 탐지 후 XGROUP DESTROY
  • 모니터링 지표
    • Pub/Sub: 채널별 수신율, 처리 지연, 파드 수 변동 대비 누락 여부
    • Streams: Pending 수, ACK 지연, 메모리 사용량, XADD 실패율
  • 보존·용량. XTRIM 정책과 만료 기준 수립, 장애 시 재처리 창 고려

마무리#

요약

  • fan-out으로 병렬 분산, fan-in으로 결과 집계와 완료 판정 수행
  • SNS는 고병렬 브로드캐스트에 단순하고 빠름. 멱등성 확보 필요
  • SQS는 경쟁 소비자 모델, 가시성 타임아웃·DLQ·백프레셔로 제어 용이
  • Kinesis·DynamoDB Streams는 샤드 기반 병렬성 제약이 있으나 순서 보장이나 CDC 파이프라인에는 적합
  • 쿠버네티스 내부에서 현재 파드 전부에 전달 보장이 핵심이면 Redis Streams + 파드별 Consumer Group 권장. 단순 저지연이면 Pub/Sub 선택

한줄 정리

  • 대량 병렬화는 SNS/SQS로, 강한 전달 보장은 SQS·Streams 계열로, intra-cluster 전송은 Redis 구성이 현실적 선택지임
