NestJS forRoot는 싱글톤 보장 아님, global true로는 중복 호출 못 막음

December 9, 2025

개요#

forRoot 패턴이 자동으로 싱글톤을 보장하지 않음 동일 모듈에서 forRoot를 여러 번 호출하면 호출 횟수만큼 프로바이더 인스턴스가 만들어짐 @nestjs/schedule의 ScheduleModule도 예외 아님

핵심 개념#

  • dynamic module forRoot
    • forRoot는 DynamicModule을 반환하는 팩토리 패턴
    • 호출할 때마다 새로운 동적 모듈 토큰 생성, 별개의 모듈 인스턴스로 취급됨
  • global true의 의미
    • 모듈을 전역 스코프로 노출하여 재임포트 없이 프로바이더 주입 가능하게 함
    • 전역 노출일 뿐, 중복 생성 방지 수단은 아님

간단 스니펫

static forRoot(...) {
  return {
    global: true,
    module: ScheduleModule,
    providers: [ScheduleExplorer, SchedulerRegistry],
    exports: [SchedulerRegistry],
  }
}

동작 원리#

  • ScheduleExplorer는 OnModuleInit 훅에서 애플리케이션 프로바이더를 스캔하고 @Cron 메서드를 등록함
  • forRoot를 두 번 호출하면 ScheduleExplorer 인스턴스가 2개 생성됨
  • 각 인스턴스가 같은 @Cron 메서드를 각각 등록하므로 작업이 중복 등록됨

개념 스니펫

class ScheduleExplorer implements OnModuleInit {
  onModuleInit() {
    // @Cron 메서드 탐색 및 등록
  }
}

왜 중복이 발생하는가#

  • 정적 모듈은 프레임워크 레벨에서 중복 임포트를 통합하는 경향이 있음
  • 반면 forRoot가 반환하는 동적 모듈은 호출 입력을 바탕으로 고유 토큰이 생김
  • 같은 모듈 클래스를 대상으로 해도 forRoot를 두 번 부르면 서로 다른 동적 모듈로 취급되어 프로바이더도 두 벌 생성됨

사례와 증상#

  • 서로 다른 피처 모듈에서 각각 ScheduleModule.forRoot를 호출한 경우
  • 동일한 @Cron 메서드가 두 번 이상 등록되어 주기가 배로 실행되는 증상 발생

예시 스니펫

// 루트 모듈
@Module({ imports: [ScheduleModule.forRoot()] })
export class AppModule {}

// 피처 모듈에서 또 호출하면 중복 등록 위험
@Module({ imports: [ScheduleModule.forRoot()] })
export class FeatureModule {}

주의와 베스트프랙티스#

  • forRoot는 애플리케이션 루트에서 한 번만 호출하는 관례 준수
  • 전역 모듈로 노출되어 있으면 하위 모듈에서는 재호출 없이 주입만 사용
  • 라이브러리 수준에서 중복 호출을 막고 싶다면 내부적으로 가드 플래그를 두어 재호출 시 에러를 던지는 방식 채택 가능
  • @nestjs/schedule은 이런 가드가 없으므로 호출 위치와 횟수 관리 필요

체크리스트#

  • 예상보다 작업이 더 자주 실행됨 → forRoot 중복 호출 여부 점검
  • 같은 프로바이더가 모듈 초기화 로그를 두 번 출력함 → 동적 모듈 중복 인스턴스 생성 가능성 높음

결론#

  • 질문
    • forRoot가 싱글톤을 보장하는가
  • 답변
    • 보장하지 않음, global true만으로는 중복 호출을 막지 못함
  • 원인
    • 동적 모듈을 호출 횟수만큼 생성, ScheduleExplorer 같은 라이프사이클 훅이 중복 실행되어 @Cron이 여러 번 등록됨
  • 해결
    • forRoot를 루트에서 한 번만 호출, 하위 모듈은 재호출 금지
