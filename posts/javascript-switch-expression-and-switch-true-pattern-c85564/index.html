JavaScript에서 switch를 표현식처럼 쓰는 방법과 switch(true) 패턴

November 27, 2025

개요#

조건 분기에서 switch는 여전히 많이 쓰이는 구문임. 다만 switch는 문(statement)라서 값을 곧바로 반환하기엔 번거로운 경우가 있음. 이 글은 두 가지 실용 패턴을 정리함. 객체 매핑으로 switch를 표현식처럼 다루는 방식과, 여러 불리언 조건을 깔끔하게 열거하는 switch(true) 패턴임. 둘 다 기존 if/else 또는 전통적 switch를 대체하는 선택지로 가독성과 유지보수에 이점이 있는 케이스가 존재함

전통적 switch vs 객체 매핑#

전통적 switch는 명령형으로 흐름을 기술하는 데 적합함. 그러나 단순 키-값 대응이라면 객체 매핑이 더 짧고 의도를 드러냄

전통적 switch의 전형적 형태

let result
switch (key) {
  case 'A':
    result = 'resultA'
    break
  case 'B':
    result = 'resultB'
    break
  default:
    result = 'resultDefault'
}

동일 로직을 객체 매핑으로 치환하는 예시

const result = ({ A: 'resultA', B: 'resultB' })[key] ?? 'resultDefault'

핵심 포인트

  • 단순 매핑은 객체 조회로 충분. 분기 수가 늘어도 구조 고정
  • 기본값 처리에 || 대신 ?? 권장. 매핑 값이 빈 문자열이나 0처럼 falsy일 때 오작동 방지
  • 키가 런타임 문자열로 안전하게 제한되는 경우 적합. 동적 계산 로직이 필요한 경우엔 부적합
  • 프로토타입 체인 오염 여지 차단 위해 리터럴 객체 사용 권장. 필요 시 Map으로 대체 가능

switch(true) 패턴의 개념#

switch는 값을 비교하지만, 조건식도 평가식이므로 true를 피봇으로 쓰면 케이스마다 불리언 표현식을 배치할 수 있음. 첫 번째로 true가 되는 케이스가 실행됨

switch (true) {
  case 1 + 1 === 2:
    // 실행됨
    break
  default:
    // 미실행
}

이 패턴의 의도

  • 여러 독립 조건을 상하로 정렬해 가독성 확보
  • if/else 사슬보다 각 조건을 시각적으로 블록화
  • OR가 많은 복잡 조건을 여러 줄 케이스로 분해 가능

언제 유용한가#

  • 유효성 검증처럼 실패 조건을 위에서부터 빠르게 탈출해야 하는 흐름
  • 조건 간 결합도가 낮고, 첫 만족 조건에서 즉시 반환 또는 예외를 던지는 패턴
  • 긴 OR 조건을 줄 단위로 나눠 리뷰 난이도와 변경 용이성을 개선해야 하는 경우

간단 검증 흐름 예시

switch (true) {
  case !user:
    throw new Error('user required')
  case typeof user.firstName !== 'string':
    throw new Error('firstName must be string')
  default:
    return user
}

복잡 OR 조건을 케이스로 분리하는 예시

switch (true) {
  case user.age > 40:
  case user.name.startsWith('B'):
  case user.name === 'Sam' && user.age === 31:
    return `Welcome ${user.name}!`
  default:
    return 'Welcome unknown user'
}

동작 원리와 구조#

  • switch는 전달된 표현식과 각 case 표현식의 평가 결과를 엄격 동등비교로 대조함
  • switch(true)에서는 각 case가 불리언으로 평가됨. true와 일치하는 첫 케이스 실행
  • break 또는 return으로 블록 종료 필요. 누락 시 다음 케이스로 폴스루 발생

주의할 점과 트레이드오프#

  • 팀 컨벤션. switch(true)는 호불호가 존재. 낯설어 가독성 논란 발생 가능
  • 남용 시 if/else와 다르지 않음. 5~7개 이상 조건이 쌓이면 추상화 또는 전략 패턴 고려
  • 케이스 표현식은 순서 의존. 상호 배타성이 보장되지 않으면 상단 조건이 하단 조건을 가릴 수 있음
  • 부수효과 금지 권장. 케이스 내 표현식은 판단만 수행. 상태 변경은 블록 내부로 한정
  • 디버깅 가독성. 케이스에 이름이 없어 의도를 드러내기 어려울 수 있음. 가능하면 isXxx 같은 헬퍼로 의도 명시
  • 값 매핑과 행위 분기 구분 필요. 값 대응은 객체 매핑 또는 Map, 로직 분기는 switch(true) 혹은 if/guard가 적합

베스트 프랙티스#

  • 매핑 기반 패턴
    • 값 조회에는 객체 리터럴 또는 Map 사용
    • 기본값은 ?? 사용. falsy 값 보존
    • 키 스페이스 제한 및 스키마 검증 권장
  • switch(true) 패턴
    • 케이스는 순서가 중요. 구체적이고 엄격한 조건을 상단 배치
    • 케이스 표현식은 순수 함수 기반 헬퍼로 분리해 의도 명확화
    • 각 케이스는 즉시 반환 또는 예외로 종료. fall-through는 의도적일 때만 사용
    • 케이스 수가 늘면 규칙 테이블화 또는 전략 객체로 승격 고려

짧은 비교 가이드#

  • 단순 키→값 매핑 필요. 객체 매핑 사용
  • 동적 계산 결과에 따라 다른 로직 실행. switch(true) 또는 guard if 사용
  • 많은 OR를 읽기 쉬운 다중 라인으로 분리. switch(true) 후보
  • 데이터 중심 규칙이 늘어남. 규칙 테이블화 또는 DSL 검토

마무리#

switch를 반드시 버릴 이유는 없지만, 값을 선택하는 문제는 매핑이 더 단순함. 다수의 독립 조건을 선형으로 평가해 빠르게 탈출해야 하는 문제는 switch(true)가 유효한 대안이 됨. 팀 합의와 코드베이스 일관성을 우선하며, 값 매핑과 행위 분기를 명확히 구분해 적용하는 것이 핵심임
