Solana 아키텍처와 핵심 기술 개념 정리

October 10, 2025

개념과 배경#

Solana는 처리량과 비용 효율을 우선한 퍼블릭 블록체인으로, 스마트 컨트랙트와 디앱 실행에 맞춘 고성능 설계를 채택함 목표는 낮은 지연, 높은 TPS, 저렴한 수수료를 동시에 달성하는 것임 이를 위해 시간 합의 보조 기법과 병렬 실행 런타임, 네트워크 전파 최적화를 결합하여 기존 블록체인의 병목을 줄이는 방향으로 진화해옴

핵심 개념과 정의#

  • Proof of History PoH
    • 연속 해시를 이용해 시간 순서를 암호학적으로 증명하는 기법
    • 트랜잭션 도착 순서와 이벤트 시퀀스를 사전 정렬하는 효과를 제공, 합의 과정의 통신 오버헤드 감소
  • Proof of Stake PoS와 Tower BFT
    • 스테이크에 기반한 검증자 선택과 투표로 합의 진행
    • PoH가 제공하는 글로벌 시계에 기대어 빠른 파이널리티를 지향하는 BFT 스타일 합의 운영
  • Sealevel
    • 계정 락 기반의 병렬 실행 엔진
    • 트랜잭션이 접근하는 계정을 명시하도록 강제하여 충돌 없는 작업을 동시 실행
  • Gulf Stream
    • 멤풀을 최소화하고 차기 리더에게 트랜잭션을 선제적으로 전달
    • 리더 슬롯 시작 즉시 실행 가능 상태 확보, 대기 시간 단축
  • Turbine
    • 데이터 전파를 소형 패킷으로 샤딩해 계층적으로 확산
    • 대역폭 사용 최적화와 패킷 손실 내성 강화
  • Pipelining, Cloudbreak
    • 블록 생산과 검증 파이프라인 분리, 저장소 레이아웃 최적화로 I/O 병목 완화
  • 네트워크 및 수수료 모델
    • QUIC 기반 전송 채널과 로컬 수수료 시장, 우선순위 수수료로 혼잡 구간에서 성능 저하 완화

동작 원리와 구조#

  • 리더 스케줄과 슬롯
    • 스테이크 가중치에 따라 검증자가 슬롯 단위로 리더 역할을 수행, 트랜잭션을 블록에 포함
    • 투표 트랜잭션으로 블록 체인을 확정하고 보상 분배
  • 계정 기반 프로그래밍 모델
    • 프로그램은 불변 코드, 계정은 상태 저장소 역할
    • 트랜잭션은 접근할 계정 목록과 읽기/쓰기 권한을 사전에 선언, 런타임이 계정 충돌을 감지해 병렬화 결정
  • 병렬 실행의 성패 포인트
    • 계정 충돌이 적을수록 Sealevel의 병렬성이 극대화됨
    • 동일한 인기 계정을 다수 트랜잭션이 동시에 수정하려 하면 직렬화 발생
  • 리소스와 한도
    • 트랜잭션 크기 제한, 컴퓨트 유닛 한도, 서명 수 제한 등 존재
    • 우선순위 수수료를 통해 혼잡 시 처리 순위를 조정 가능

주요 사용 사례#

  • 디파이
    • 고빈도 거래와 주문 체결 같은 지연 민감 워크로드에 유리한 환경
  • NFT 및 디지털 자산
    • 낮은 수수료와 빠른 민팅·전송 경험
  • 게임 및 실시간 애플리케이션
    • 짧은 확정 시간을 요구하는 상호작용형 시나리오에 적합
  • 토큰 발행
    • 표준 라이브러리 기반의 토큰 정의, 전송, 메타데이터 관리 지원

다른 블록체인과의 비교#

아래 값은 네트워크 상태, 클라이언트 구현, 하드웨어 조건에 따라 달라질 수 있는 대표 지표로 이해할 것

특징Solana이더리움 Ethereum폴리곤 Polygon
합의 알고리즘PoH + PoS, Tower BFTPoSPoS
처리 속도고TPS 병렬 실행 전제수십 TPS 레이어1 기준수천 TPS 레이어2
수수료매우 낮음변동성 높음낮음
스마트 컨트랙트Rust C C++ 기반 프로그램Solidity 중심Solidity 중심
확장성높은 병렬화샤딩 L2로 확장사이드체인 확장

주의 사항과 한계, 베스트 프랙티스#

  • 중앙화 논쟁
    • 높은 하드웨어 요구와 대역폭 필요로 검증자 진입 장벽 존재
    • 클라이언트 다양성과 검증자 분포가 장기 신뢰성에 중요
  • 서비스 중단 이력
    • 특정 패턴의 트래픽 폭주, 네트워크 레이어 이슈가 과거에 중단으로 이어진 사례 존재
    • 릴리스 업그레이드와 가드레일 도입으로 점진 개선 중
  • 개발 시 유의점
    • 계정 충돌 최소화 설계 필수, 쓰기 세트를 분리한 상태 모델링 권장
    • 컴퓨트 유닛 예산과 우선순위 수수료를 명시해 혼잡 구간 지연 완화
    • 트랜잭션 재전송 전략, 블록해시 만료 관리, 실패 사유 로깅 체계화 필요
  • 트레이드오프
    • 병렬성과 낮은 수수료를 위해 복잡한 런타임 제약과 네트워크 요구가 증가하는 구조

간단 예시#

혼잡 시 우선순위 수수료와 컴퓨트 예산을 설정해 처리 안정성 확보하는 패턴 예시

// pseudo TypeScript using a generic SDK
const ix1 = makeSetComputeUnitLimit({ units: 600000 })
const ix2 = makeSetComputeUnitPrice({ microLamports: 5000 }) // priority fee
const ix3 = makeProgramInstruction({ accounts, data })

const tx = new Transaction()
  .add(ix1)
  .add(ix2)
  .add(ix3)

await sendAndConfirm(tx, { maxRetries: 3 })

마무리#

Solana는 PoH 기반의 글로벌 시계와 Sealevel 병렬 실행, 네트워크 전파 최적화를 결합해 고성능을 달성하려는 접근을 취함 개발자는 계정 충돌을 줄이는 데이터 모델링과 리소스 한도 관리, 우선순위 수수료 활용을 통해 성능 이점을 실전에 연결할 수 있음 장점인 처리량과 비용 효율을 활용하되 검증자 분산과 네트워크 안정성 이슈를 모니터링하며 아키텍처 결정을 내리는 것이 현실적인 선택임
