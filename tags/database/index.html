개요 InnoDB 버퍼 풀은 MySQL InnoDB 스토리지 엔진의 메모리 캐시 영역으로 데이터와 인덱스 페이지를 담아 디스크 I/O를 줄이고 지연 시간을 낮추는 역할 수행 InnoDB는 트랜잭션, 로우 레벨 락, 복구 기능을 제공하며 버퍼 풀은 그 성능의 핵심 구성요소임 핵심 개념 페이지 캐싱: 테이블 데이터와 인덱스 페이지를 메모리에 유지, 기본 16KB 페이지 단위 운용 쓰기 지연: 변경된 페이지는 메모리에서 더티 페이지로 남고 백그라운드 플러시로 디스크 반영 트랜잭션 로그: 변경 내용은 먼저 리두 로그에 기록되어 크래시 시 일관성과 복구 보장 LRU 교체: Least Recently Used 기반으로 오래 사용되지 않은 페이지부터 축출하여 자주 쓰는 페이지 유지 동작 원리 읽기 경로: 요청한 페이지가 메모리에 있으면 히트, 없으면 디스크에서 로드 후 버퍼 풀에 적재 쓰기 경로: 커밋 시 로그 동기화 우선, 데이터 페이지는 지연 플러시로 배치 처리하여 디스크 쓰기 부담 완화 교체 정책: 메모리 압박 시 LRU에서 덜 사용된 페이지 제거, 워크로드에 따라 캐시 효율 크게 좌우됨 크기 설정과 운영 기준 innodb_buffer_pool_size는 워크로드와 시스템 메모리 여유를 기준으로 결정 단일 데이터베이스 서버에서는 전체 RAM의 큰 비중을 할당하는 경우가 많으나 운영체제 캐시와 공존 프로세스를 감안해 스왑 발생을 피하는 선 유지 권장 관찰 지표: 버퍼 풀 히트 비율, 더티 페이지 비율, 플러시 대기 시간 등으로 튜닝 효과 검증 최신 버전에서는 버퍼 풀 크기 동적 조정 지원, 부하 패턴 변화에 맞춘 점진적 조정 권장 정리 적절히 설정된 InnoDB 버퍼 풀은 읽기 히트율 상승과 쓰기 배치 처리로 일관된 성능을 제공함 메모리 제약과 워크로드 특성을 기반으로 보수적으로 시작하고 관측 지표를 바탕으로 점진적 조정 권장 ...
